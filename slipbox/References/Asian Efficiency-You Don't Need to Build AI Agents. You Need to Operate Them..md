# You Don't Need to Build AI Agents. You Need to Operate Them.

> Asian Efficiency — Thanh Tran

## Resumo

A objeção mais comum em workshops de AI é "não sou técnico o suficiente". Thanh argumenta que **o profissional mais valioso numa organização AI não é o builder — é o operator**: quem roda o sistema, alimenta os inputs certos, avalia outputs, pega erros, faz os juízos que a AI não pode fazer.

## Builder vs. Operator

| Builder | Operator |
|---|---|
| Projeta e constrói os sistemas | Roda os sistemas |
| Entende APIs, escreve prompts, configura tools | Avalia outputs, pega erros, decide quando intervir |
| Conhecimento técnico profundo | Conhecimento profundo do domínio |
| Habilidade real, leva tempo para desenvolver | **É o que escala** |
| Para cada builder, precisa de múltiplos operators | |

O operator não precisa saber programar. Precisa conhecer o trabalho. Tem o domínio que o builder tipicamente não tem.

## Como funciona na prática

A pipeline de conteúdo da AE (transcripts → story database → múltiplos passos de escrita → formatação por canal) foi construída por Thanh. Quem roda é a OBM — entende o que é um bom post para cada plataforma, sabe quando a AI capturou o ângulo certo, pega erros factuais, reconhece quando a voz está off, faz decisões editoriais que o sistema não consegue fazer sozinho.

Mesma lógica de gerir um funcionário capaz: não precisa entender todos os aspectos técnicos do background dele para dar direção efetiva — precisa entender o trabalho bem o suficiente para avaliar o output, dar feedback útil e redirecionar quando algo sai do trilho.

## A habilidade que vale desenvolver

> "If you're waiting to get 'technical enough' before engaging with AI at work, you're waiting for the wrong thing."

Os operators que criam mais alavancagem não são necessariamente os melhores prompt engineers. São as pessoas que entendem o domínio profundamente — clientes, indústria, barra de qualidade do trabalho — e aplicam esse conhecimento para avaliar e direcionar o output da AI.
